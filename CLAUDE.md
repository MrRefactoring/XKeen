# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Что это

XKeen — утилита управления прокси-клиентом (Xray/Mihomo) на роутерах Keenetic под Entware. **Только POSIX `sh`**, компилируемого кода нет. Репозиторий содержит исходники и упаковку; исполняется всё на роутере.

Это **форк** `jameszeroX/XKeen`. Работа ведётся как над самостоятельным проектом: ветки уходят в `origin` (`MrRefactoring/XKeen`) и мержатся в его же `main` через PR. Пуш в `upstream` заблокирован намеренно (`git remote` показывает `DISABLED`).

Целевые архитектуры: `arm64-v8a`, `mips32le`, `mips32`.

## Engineering Principles

Применять ко **всему**, что пишется в репозитории — код, тесты, скрипты, конфигурация.

- **KISS** — самое прямое решение, которое работает. Никаких трюков там, где хватает простого кода. В POSIX sh это особенно важно: `eval`, вложенные подстановки и хитрые перенаправления читаются плохо и ломаются молча.
- **YAGNI** — только то, что требует текущая задача. Никаких умозрительных опций, абстракций и «задела на будущее». Функция, которую никто не вызывает, — мёртвый код, а не API.
- **DRY** — не дублировать знание. Выносить общее, когда **та же** логика появилась в нескольких местах. Но не пересушивать: две внешне похожие строки, которые могут разойтись, — не дублирование. При конфликте побеждают KISS и YAGNI.
- **SOLID**, применительно к shell:
  - **S** — одна ответственность. Функция делает то, что обещает имя, и не больше. Разбор данных и их вывод — разные функции.
  - **O** — расширять новым кодом, а не правкой устоявшегося. Новая команда — ветка `case` в диспетчере и новый модуль, а не переписывание существующих.
  - **L** — взаимозаменяемость. Валидаторы для `ask_value` и для `fetch_with_mirrors` обязаны соблюдать один контракт: код возврата и набор выставляемых переменных, — иначе подстановка одного вместо другого сломает вызывающего.
  - **I** — узкий интерфейс. Не заставлять вызывающего передавать то, что он не использует; не плодить обязательные аргументы ради редкого случая.
  - **D** — зависеть от абстракций проекта, а не от примитивов:

    | Нужно | Использовать | Не использовать напрямую |
    | --- | --- | --- |
    | Ввод от пользователя | `ask_one` / `ask_many` / `ask_value` / `ask_yesno` | `read` |
    | Загрузка с GitHub | `fetch_with_mirrors`, `probe_with_mirrors` | `curl` |
    | Сетевой запрос с таймаутом | `curl_with_timeout`, `curl_api` | `curl` |
    | Пути и URL | переменные из `01_info_variable.sh` | литералы `/opt/...`, `https://github.com/...` |
    | Очистка экрана | `smart_clear` | `clear` |

При конфликте принципов выигрывает простота: KISS и YAGNI важнее преждевременной структуры ради SOLID/DRY.

## Проверки

Целевая среда — BusyBox ash, поэтому проверять надо в ней, а не в bash. **`dash` не подходит**: в нём нет `read -p`, который используется по всему проекту.

```sh
# Синтаксис всего дерева
docker run --rm -v "$PWD:/repo:ro" busybox:1.36 sh -c '
  for f in $(find /repo/scripts -name "*.sh") /repo/scripts/xkeen; do sh -n "$f" || echo "СБОЙ: $f"; done'

# Тесты
docker run --rm -v "$PWD:/repo:ro" busybox:1.36 sh -c '
  for t in test_ask test_ask_keys test_mirrors; do sh /repo/spec/$t.sh | tail -1; done'

# Один тест
docker run --rm -v "$PWD:/repo:ro" busybox:1.36 sh /repo/spec/test_ask.sh

# Тесты, которым нужен jq (alpine — та же BusyBox ash, но с пакетами)
docker run --rm -v "$PWD:/repo:ro" alpine:3.19 sh -c 'apk add -q jq && sh /repo/spec/test_screen_mode.sh'

# Линтер: нулевая толерантность к НОВЫМ предупреждениям, фон в базе большой
shellcheck -s sh scripts/xkeen $(find scripts -name '*.sh') | grep -oE 'SC[0-9]+' | sort | uniq -c | sort -rn
```

CI (`.github/workflows/checks.yaml`) гоняет синтаксис и тесты жёсткими воротами, ShellCheck — отчётом.

## Прогон на живом роутере

Контейнер ловит не всё: у Entware свой набор аплетов BusyBox. Реальные расхождения, на которых уже падали тесты:

- **нет `timeout`** — сторожи в тестах писать через `sleep` + `kill`;
- **нет `pkill`** — процессы искать через `ps w` + `awk`;
- **`date +%s%N` не работает** — наносекунд нет, мерить повторами;
- **нет `sftp-server`** — `scp` не работает, файлы передавать через `tar`;
- `/tmp` — tmpfs, `/opt` — раздел на диске.

```sh
tar -czf - --exclude='._*' spec scripts | ssh giga '
  rm -rf /opt/tmp/repo; mkdir -p /opt/tmp/repo && cd /opt/tmp/repo && tar -xzf - \
  && find . -name "._*" -delete && sed -i "s|/repo/|/opt/tmp/repo/|g" spec/*.sh
  for t in test_ask test_ask_keys test_mirrors; do sh spec/$t.sh | tail -1; done'
```

## Ловушки, на которых легко потерять время

**`_xkeen` в репозитории — это `.xkeen` на роутере.** `install_xkeen_rename` переименовывает каталог при первом запуске. Все runtime-пути ссылаются на `.xkeen`, все пути в репозитории — на `_xkeen`.

**`$(...)` — субшелл.** Значение, выставленное внутри, не переживает возврат; читать терминал оттуда нельзя. Из этого следует:
- функции слоя диалога возвращают результат через глобальные `REPLY_KEY` / `REPLY_KEYS` / `REPLY_VALUE`, а не через stdout;
- `return $(func)` подставит **текст**, а не код возврата;
- `$(printf '\n')` срезает завершающий перевод строки;
- в тестах ввод подавать here-doc или редиректом, но не пайпом.

**`set -o pipefail` не существует в POSIX sh.** `$?` после пайпа — код последнего элемента. Для кода `curl` в `curl_with_timeout` используется отдельный дескриптор.

**Ввод — только через `ask_*`** (`scripts/_xkeen/04_tools/05_tools_choice/00_choice_ask.sh`), напрямую `read` не писать. Подробности — в `docs/contributing.md`.

**Без TTY `read` сразу получает EOF.** Цикл перезапроса на пустых строках крутится бесконечно и жжёт процессор. Функции `ask_*` возвращают ненулевой код — его надо проверять.

**Навигация стрелками зависит от сборки BusyBox.** `read -n`/`-s` — опции времени сборки, поэтому поддержка определяется в рантайме с откатом на ввод по номерам.

**`echo -e` в проекте повсеместен** (468 вхождений) и корректно работает в BusyBox ash. Массово менять на `printf` не нужно.

## Пути и URL

Все определены в `scripts/_xkeen/01_info/01_info_variable.sh` — единственное место. Не хардкодить ни `/opt/...`, ни `https://github.com/...` в других файлах.

## Каталоги `test/` и `spec/` — разное

- `test/` — артефакт dev-канала, туда CI коммитит `xkeen.tar.gz`. Руками не трогать.
- `spec/` — юнит-тесты. В релизный архив не попадают: `package-folder.yaml` пакует только `scripts/*`.
